4 /*This basic image class will only support floating point images
5  * (a must these days) composed of three channels only. You can easily
6  * extend this class as an exercise, for example you can use template to
7  * define the image data type and the number of channels on the fly..
8  *
9  * The image resolution defines the dimension of this 2D array.
10  * The width (horizontal dimension) and the height (vertical dimension)
11  * define the overall resolution of the image. A pixel in the digital world
12  * is generally defined by three digital values, one for each elementary or
13  * primary color.
14  *
15  * The number of bits (not bytes) used per pixel is often abbreviated with
16  * the acronym bpp; it stands for bits per pixel. Truecolor uses 24 bpp.
17  * Sometimes the number of bits used is defined per channel (i.e. per color).
18  * In this case the acronym is bpc (bits per channel) but it is not often used.
19  * The number of bits used for each color component of a single pixel,
20  * is known as the color depth or bit depth.
21  *
22  * Storing pixel color as floats was originally motivated by the need for
23  * high dynamic range images (or HDR images), images capable of storing the
24  * wider range of colors and light levels one can observe in the physical
25  * world.
26  *
27  * From a programming point of view, all we need to do, is to create a
28  * simple Image class, in which we will store the width and the height of
29  * the image, as well as a 1D array of pixels (you can use a two dimensional
30  * arrays if you want, but this is not necessarily more practical and is
31  * not as efficient). We need two constructors, one to create an empty
32  * image and one to create an image with a user specified resolution (line 20)
33  * which can also be filled with a given constant color (this color default
34  * to black).
35  *
36  * The pixels themselves are represented with a special structure named Rgb
37  * (line 5). This is very convenient because we can write operators to
38  * manipulate the three floats of a pixel at once. It is trivial to access
39  * pixels from the image by overloading the bracket operator []
40  * (line 25 and 26) and applying every mathematical operations we want on
41  * these pixels: multiplying them (line 11), comparing them (line 10),
42  * adding them (line 12), etc. The function (line 13) is interesting.
43  * It can be used to compute the brightness of a pixel and accumulate
44  * the result into a float. It will be used in the next lesson.
45  *
46  * Finally pixels are stored in the image as a 1D array of Rgb elements
47  * (line 29). The size of this area is equal to the width of the image
48  * multiplied by the image height. It is not necessary to multiply this
49  * number by the number of channels since each pixel (of type Rgb) already
50  * holds the memory to store three colors.
51  *
52  * References:
53  * http://www.scratchapixel.com/
54  * http://www.scratchapixel.com/old
55  */

203 /* readPPM :
204  *
205  * The header (the few lines at the top of the file providing the necessary
206  * information about the image we are about to read) is defined by three lines.
207  * The first line contains a string which can either be "P1", "P2", ... "P6".
208  * We are only concerned by "P6" PPM images which stands for binary color
209  * images. The next line contains two integers representing the resolution
210  * of the image. The third line contains an another integer representing
211  * the number of levels each color is encoded with (i.e. the number of grey
212  * values between black and white, generally 255). You can insert comments
213  * in the header; they start with the letter '#', but we will ignore them
214  * in this lesson.
215  *
216  * In the PPM format, colors are encoded using 24 bpp. In other worlds each
217  * color is encoded as 1 bytes (in C++ an unsigned char). Each pixel is
218  * encoded in the file as a sequence of three bytes, one for each
219  * color.
220  *
221  * Pixels are encoded in scanline order, from the first row to the last row
222  * in the image, for left to right. In other words the first three
223  * bytes in the file are for the pixel at location (1,1) in the image
224  * (where (1,1) denotes the upper left corner of the image), and the last
225  * three bytes in the file represent the lower right pixel in the frame
226  * (location (640, 480) assuming the image has resolution 640x480).
227  */
228 Image readPPM(const char *filename)
229 {
230  std::cout<<"Reading image: "<<filename<<std::endl ;
231  std::ifstream ifs;
232  ifs.open(filename, std::ios::binary); // need to spec. binary mode for Windows users
233  Image img;
234  try {
235  if (ifs.fail()) { throw("Can't open input file"); }
236  std::string header;
237  int w, h, b;
238  ifs >> header;
239  if (strcmp(header.c_str(), "P6") != 0) throw("Can't read input file");
240  ifs >> w >> h >> b;
241  img.w = w; img.h = h;
242  img.pixels = new Image::Rgb[w * h]; // this is throw an exception if bad_alloc
243  ifs.ignore(256, '\n'); // skip empty lines in necessary until we get to the binary data
244  unsigned char pix[3];
245  // read each pixel one by one and convert bytes to floats
246  for (int i = 0; i < w * h; ++i) {
247  ifs.read(reinterpret_cast<char *>(pix), 3);
248  img.pixels[i].r = pix[0] / 255.f;
249  img.pixels[i].g = pix[1] / 255.f;
250  img.pixels[i].b = pix[2] / 255.f;
251  }
252  ifs.close();
253  }
254  catch (const char *err) {
255  fprintf(stderr, "%s\n", err);
256  ifs.close();
257  }
258  return img;
259 }
